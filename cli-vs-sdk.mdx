# CLI vs SDK Comparison

Choose the approach that best fits your development workflow. The Erdo platform supports both traditional CLI tools and pure Python SDK approaches.

## Quick Comparison

<Tabs>
  <Tab title="CLI (Traditional)">
    **Best for:** Command-line workflows, CI/CD pipelines, quick setup

    **Pros:**
    - Quick installation with Homebrew
    - Familiar command-line interface
    - Great for scripts and automation
    - Works with any editor
    
    **Cons:**
    - Requires separate Go binary installation
    - Less IDE integration
    - Context switching between Python and CLI
  </Tab>
  <Tab title="SDK (New)">
    **Best for:** Python-first workflows, Jupyter notebooks, programmatic control

    **Pros:**
    - Pure Python - no external dependencies
    - Excellent IDE integration and type hints
    - Programmatic control over all operations
    - Perfect for notebooks and interactive development
    
    **Cons:**
    - Newer approach (less battle-tested)
    - Python-only (no other language support)
  </Tab>
</Tabs>

## Core Operations

### Agent Synchronization

<Tabs>
  <Tab title="CLI">
    ```bash
    # Sync all agents in current directory
    erdo sync
    
    # Sync specific agent file
    erdo sync my_agent.py
    
    # Sync with options
    erdo sync --agent my-agent --env production --dry-run
    
    # Sync from directory
    erdo sync --exclude "*.log,temp/*"
    ```
  </Tab>
  <Tab title="SDK">
    ```python
    from erdo.sync import Sync
    
    # Sync single agent
    result = Sync(agent)
    print(f"Synced: {result.agent_key}")
    
    # Sync from file
    results = Sync.from_file("my_agent.py")
    
    # Sync from directory
    results = Sync.from_directory("agents/", exclude_patterns=["*.log"])
    
    # Advanced options
    result = Sync(agent, source_file_path="agents/my_agent.py")
    ```
  </Tab>
</Tabs>

### Agent Invocation

<Tabs>
  <Tab title="CLI">
    ```bash
    # Invoke agent with message
    erdo invoke my-agent --message "analyze data"

    # With datasets and parameters
    erdo invoke my-agent \
      --message "Query" \
      --dataset sales-2024 \
      --parameter key=value

    # Test modes
    erdo invoke my-agent --message "Test" --mode replay  # Free

    # Output formats
    erdo invoke my-agent --message "Query" --output text
    erdo invoke my-agent --message "Query" --output json

    # Streaming
    erdo invoke my-agent --message "Long task" --stream --verbose
    ```
  </Tab>
  <Tab title="SDK">
    ```python
    from erdo import invoke

    # Invoke agent
    response = invoke(
        "my-agent",
        messages=[{"role": "user", "content": "analyze data"}],
        parameters={"query": "test"}
    )

    if response.success:
        print(response.result)

    # With datasets
    response = invoke(
        "my-agent",
        messages=[{"role": "user", "content": "Query"}],
        datasets=["sales-2024"],
        mode="replay"  # Free after first run
    )

    # Streaming mode
    response = invoke(
        "my-agent",
        messages=[{"role": "user", "content": "Long task"}],
        stream=True,
        output_format="text"
    )
    ```
  </Tab>
</Tabs>

### Testing & Validation

<Tabs>
  <Tab title="CLI">
    ```bash
    # Run agent tests (parallel execution)
    erdo agent-test tests/test_my_agent.py

    # Verbose output
    erdo agent-test tests/test_my_agent.py --verbose

    # Local validation (static checks)
    erdo test my_agent.py

    # Test with simulation
    erdo test --data test_data.json
    ```
  </Tab>
  <Tab title="SDK">
    ```python
    from erdo import invoke
    from erdo.test import text_contains

    # Write agent tests
    def agent_test_basic_query():
        """Test basic agent invocation."""
        response = invoke(
            "my-agent",
            messages=[{"role": "user", "content": "Test"}],
            mode="replay",  # Free after first run
        )

        assert response.success
        assert text_contains(str(response.result), "expected")

    # Run tests programmatically
    from erdo.test import run_tests
    exit_code = run_tests("tests/test_my_agent.py", verbose=True)
    ```
  </Tab>
</Tabs>

## Development Workflows

### CLI-First Workflow

```bash
# 1. Setup
erdo init my-project
cd my-project

# 2. Development
erdo dev  # Start development server

# 3. Test & Deploy
erdo sync
erdo test my-agent
erdo deploy my-agent --env production
```

### SDK-First Workflow

```python
# 1. Create agent in Python
from erdo import Agent
from erdo.actions import llm
from erdo.sync import Sync
from erdo.invoke import Invoke

agent = Agent(name="my_agent")
# ... define steps ...

# 2. Sync to platform
result = Sync(agent)
print(f"Agent deployed: {result.agent_key}")

# 3. Test immediately
test_result = Invoke.by_key(
    result.agent_key,
    parameters={"test": "data"}
)
print(f"Test result: {test_result.data}")
```

### Mixed Workflow

You can combine both approaches:

```bash
# Use CLI for project setup
erdo init my-project
erdo login
```

```python
# Use SDK for development and testing
from erdo import Agent
from erdo.sync import Sync
from erdo.invoke import Invoke

# Create agent
agent = Agent(name="my_agent")
# ... define agent ...

# Sync and test with SDK
sync_result = Sync(agent)
test_result = Invoke.by_key(sync_result.agent_key, parameters={})
```

```bash
# Use CLI for final deployment
erdo deploy my-agent --env production
```

## Configuration

Both approaches use the same configuration system:

### Config File (`~/.erdo/config.yaml`)

```yaml
api:
  url: "https://api.erdo.ai"
  timeout: 30

auth:
  token: "your-token"
  organization: "your-org"

development:
  auto_save: true
  watch_mode: true
```

### Environment Variables

```bash
export ERDO_API_URL="https://api.erdo.ai"
export ERDO_AUTH_TOKEN="your-token"
export ERDO_ORGANIZATION="your-org"
```

### Programmatic Config (SDK Only)

```python
from erdo.config import Config

# Custom configuration
config = Config(
    api_url="https://custom.erdo.ai",
    auth_token="custom-token",
    organization="custom-org"
)

# Use with SDK modules
from erdo.sync import Sync
sync = Sync(config=config)
```

## Feature Comparison Matrix

| Feature | CLI | SDK | Notes |
|---------|-----|-----|-------|
| Agent Creation | ✅ Templates | ✅ Programmatic | SDK offers more flexibility |
| Agent Sync | ✅ Full featured | ✅ Python-native | Same backend, different interface |
| Agent Invocation | ✅ Testing focused | ✅ Production ready | SDK better for programmatic use |
| Testing | ✅ Comprehensive | ✅ Basic | CLI has more test runner features |
| Deployment | ✅ Multi-env | ✅ Via sync | CLI has dedicated deploy commands |
| File Watching | ✅ Built-in | ❌ Manual | CLI better for development server |
| IDE Integration | ⚠️ Limited | ✅ Excellent | SDK has full type hints |
| CI/CD Integration | ✅ Shell scripts | ✅ Python scripts | Both work well |
| Learning Curve | ⚠️ Commands to learn | ✅ Python APIs | SDK more intuitive for Python devs |

## Recommendations

### Choose CLI if you:
- Prefer command-line tools and shell scripting
- Need file watching and development server features
- Want to get started quickly with templates
- Are building CI/CD pipelines with shell scripts
- Work with multiple programming languages

### Choose SDK if you:
- Work primarily in Python
- Use Jupyter notebooks or interactive development
- Want programmatic control over agent operations
- Prefer type-safe APIs with IDE autocompletion
- Are building Python applications that manage agents

### Choose Both if you:
- Want the best of both worlds
- Have a team with mixed preferences
- Need CLI for some tasks (like templates) and SDK for others (like programmatic control)
- Are migrating from CLI to SDK gradually

## Migration Path

If you're currently using CLI and want to try the SDK:

1. **Keep using CLI for authentication**: `erdo login`
2. **Install SDK**: `pip install erdo`
3. **Start with sync**: Replace `erdo sync` with `Sync.from_file()`
4. **Add invoke**: Replace test commands with `Invoke.by_key()`
5. **Gradually adopt**: Use SDK for new development, CLI for existing workflows

The configuration and authentication are shared, so you can switch between approaches seamlessly.
